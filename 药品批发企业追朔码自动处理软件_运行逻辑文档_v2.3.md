## 一、整体流程

```
用户选择Excel文件 → 读取原始数据
    ↓
提取每行的「码包装数」列值（原始数据自带，无需查询）
    ↓
┌─────────────────────────────────────────────┐
│  判断是否所有行的"码包装数"都为 "1" +        │
│  用户是否勾选了「全是1级码（零散药品）」      │
└──────────┬──────────────────────────────────┘
           │
     ┌─────┴─────┐
     │           │
  全1级        存在非1级码
  模式          混装模式
     │           │
     ▼           ▼
┌──────────┐  ┌──────────────────────────┐
│ 第1轮：   │  │ 第1轮：并发查所有码的批号  │
│ 并发查    │  │     ↓                    │
│ 所有码的  │  │ 按原始表"码包装数"分流     │
│ 批号     │  │  ├─ =1 → 1级码，批号回填  │
│ ↓         │  │  └─ ≠1 → 非1级码        │
│ 批号回填   │  │     ↓                    │
│ ↓         │  │ 第2轮：按批号分组         │

│  合并输出：码包装数统一为 1                 │
│  扫描时间：全1级模式复用第一个 / 混装模式继承 │
│  文件名：{单位名}-{收货单位}-{药品名}&...-{单据号}.xlsx │
└────────────────────────────────────────────┘
```

## 二、全 1 级模式（所有行码包装数都为 1）

### 触发条件

输入 Excel 中每一行的"码包装数"列值都是 `"1"`，且（可选）用户在 UI 勾选了 **「全是 1 级码（零散药品）」**。

> **为什么需要勾选？** 即使码包装数全为 1，用户仍可选择走混装模式（如果怀疑数据有误）。勾选是"主动确认"，降低误判概率。

### 处理流程

**2.1 操作前准备**

1. 弹窗显示**显眼警告**（红底白字）：

```
╔═══════════════════════════════════════════════╗
║  ⚠️  电脑即将自动进行操作                     ║
║                                               ║
║     请不要移动鼠标或敲击键盘！                  ║
║     否则可能导致操作失败或数据错误！             ║
║                                               ║
║     按「确定」开始，按「取消」停止               ║
╚═══════════════════════════════════════════════╝
```

2. 倒计时 5 秒后自动激活码上放心客户端窗口（`pyautogui.getWindowsWithTitle` + `.activate()`）

3. 如果窗口激活失败（窗口未打开或不存在），弹窗提示用户手动激活，等待确认后继续

**2.2 查询批号**

并发查询所有已标记为 1 级码的追溯码，仅获取「产品批号」字段：

```
for each 追溯码:
    1. 输入追溯码 → 点击查询 → 等待结果
    2. 双击批号位置 → 复制批号 → 回填
```

- 不执行「一键复制所有码」和「拆码过滤」步骤
- 如果网络超时 → 自动重试 3 次，每次间隔 2 秒
- 重试 3 次仍失败 → 弹窗报警，标记【查询失败】，用户可以"继续"或"重试"

**2.3 输出**

- 码包装数列：**保留原值 "1"**（不重新赋值）
- 扫描时间列：**复用第一个扫描时间**
- 其他列（药品器械名称、剂型规格等）：从输入表格继承

### 速度预期

假设 50 个 1 级码，每个查询约 3-5 秒 → 总耗时约 2.5-4 分钟（并发可缩短）。

## 三、混装模式（存在码包装数不为 1 的行）

### 触发条件

输入 Excel 中至少有一行的"码包装数"列值不是 `"1"`，**且**用户未勾选「全是 1 级码」。

### 第 1 轮：全量查询批号

1. 操作前准备：与全 1 级模式相同（显眼警告 + 自动激活窗口）
2. 对**所有行**的追溯码，并发查询「产品批号」字段
3. 查询结果回填到对应行

**为什么所有码都要查批号？** 批号不是原始数据自带列，必须从码上放心获取。后续的分组和回填都以批号为基础。

### 按码包装数分流

第 1 轮完成后，在内存中按原始表的「码包装数」列值分流：

| 码包装数 | 标记 | 后续处理 |
|---------|------|---------|
| = "1" | 【1 级码】 | 批号已获取，直接进入输出 |
| ≠ "1" | 【非 1 级码】 | 进入第 2 轮，查完整信息 |
| = 空白 | 【异常码】 | 重试 1-2 次 → 仍空白 → 标记异常，单独提示 |

### 第 2 轮：按批号分组查询代表码

对**非 1 级码**行：

1. 按「产品批号」分组（第 1 轮已获取）
2. 每组选 1 个代表追溯码
3. 对代表码执行**完整查询流程**：
   ```
   1. 输入追溯码 → 点击查询 → 等待结果
   2. 点击"一键复制所有码"
   3. 读取剪贴板 → 过滤 1 级码列表
   4. 获取产品批号、生产厂家、有效期等信息
   ```
4. 代表码的完整查询结果 → 回填给**同组其他非 1 级码行**
5. 如果网络超时 → 自动重试 3 次 → 弹窗报警

**为什么要按批号分组？** 同批号的非 1 级码在码上放心返回的完整信息（厂家、有效期、子码列表）一致，查 1 个=查 N 个。

### 输出

- 1 级码行：直接从第 1 轮结果输出
- 非 1 级码行：第 2 轮代表码结果回填
- 码包装数列：统一为 `"1"`（拆码后的 1 级码）
- 扫描时间列：从输入表格继承（与对应非 1 级码行一致）

### 列定义（10 列）

| 列号 | 列名 | 数据来源 |
|------|------|----------|
| 1 | 追溯码 | 查询到的 1 级码 / 原始追溯码（跳过时） |
| 2 | 药品器械名称 | 从输入表格继承 |
| 3 | 剂型规格 | 从输入表格继承 |
| 4 | 包装规格 | 从输入表格继承 |
| 5 | 批准文号 | 从输入表格继承 |
| 6 | 生产厂家 | 从输入表格继承 |
| 7 | 码类型 | 从输入表格继承 |
| 8 | 码包装数 | **全 1 级模式**：保留原值；**混装模式**：统一为 "1" |
| 9 | 产品批号 | 从码上放心查询获取 |
| 10 | 扫描时间 | **全 1 级模式**：复用第一个；**混装模式**：从输入继承 |

### 文件命名规则

```
{单位名称}-{收货单位名称下一列}-{药品器械名称1}&{药品器械名称2}&...-{单据号下一列}.xlsx
```

- **单位名称**：UI 输入框填写，默认"药品批发企业"
- **收货单位名称下一列**：原始表"收货单位名称"列右侧单元格的值
- **药品器械名称**：原始表中所有药品名称，去重后用 `&` 连接，按出现顺序
- **单据号下一列**：原始表"单据号"行右侧单元格的值
- 非法字符（`\/:*?"<>|`）自动移除

### 样式

- 字体：等线
- 对齐：左对齐
- 边框：无
- 列宽：自适应内容

## 六、异常处理

### 6.1 网络超时 / 查询失败

| 场景 | 处理 |
|------|------|
| 查询结果未加载（2 秒后页面空白） | 等待 + 重试，最多 3 次，每次间隔 2 秒 |
| 剪贴板读取为空 | 重试复制操作 1 次，仍空 → 标记【复制失败】 |
| 批号双击无响应 | 重试双击 1 次，失败 → 标记【批号获取失败】 |

**3 次重试后的最终处理**：
1. 弹出报警对话框：显示失败码 + 错误详情
2. 提供 3 个选项：
   - **继续**：跳过此行，处理下一行
   - **重试**：再尝试 1 次
   - **停止**：中断整个处理流程
3. 已处理完成的行不受影响

### 6.2 程序崩溃 / 断点续跑（方案 C）

**按行标记进度机制**：
- 每处理完一行，在内存中记录 `processed_rows.append(row_index)`
- 如果程序中途崩溃退出，重启后：
  1. 读取上次的输出 Excel（如果存在）
  2. 从原始表中找出已处理的行
  3. **跳过已处理的行**，从未处理的行继续

**记录方式**：
- 使用一个临时文件 `_progress.json` 记录已处理的行号
- 格式：`{"processed": [0, 1, 2, 5], "total": 50}`
- 每处理完 5 行写一次磁盘（避免频繁 IO）

**为什么是方案 C 而不是真断点续跑？**
- Python 程序崩溃后无法自动恢复，需要用户重新启动
- 方案 C 在用户重启后自动续跑，实现同等效果
- 实现代码量最少，稳定性最高

### 6.3 焦点丢失

**多层防护**：
1. **操作前**：弹窗警告 + 倒计时，让用户知道即将开始
2. **操作中**：每次点击前先尝试激活码上放心窗口
3. **用户跑开的情况**：设置了 pyautogui.FAILSAFE = True，鼠标移到屏幕角落可紧急停止
4. **意外弹窗**：如果检测到异常弹窗（Windows 通知、输入法弹窗等），无法自动关闭，标记为【手动介入】

### 6.4 空白码包装数

原始表中码包装数列为空白时：
1. 重试 1-2 次（重新读取单元格，确认不是读取错误）
2. 仍为空白 → 标记【异常码】
3. 处理方式：
   - 全 1 级模式下发现异常码 → 切换为混装模式
   - 混装模式下 → 标记该行为异常，弹窗提示用户确认处理方式

**为什么选 1（重试+异常码）？**
- 原始平台导出的数据空白率 < 1%，重试即可解决
- 避免将网络异常误判为"非 1 级码"
